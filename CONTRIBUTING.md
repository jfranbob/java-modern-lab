# Contributing to Java Modern Lab

First of all, thank you for your interest in contributing to **Java Modern Lab**.

This project is built around a simple idea:

> **Understand first. Measure second. Share forever.**

The goal is not to collect code examples, but to build a reliable engineering knowledge base for the Java community.

Every contribution should improve the quality, accuracy, and educational value of the project.

---

# Before You Start

Please read the following documents before contributing:

* Project Vision
* Roadmap
* Coding Standards
* Quality Gates
* Code of Conduct

Understanding the project's philosophy is more important than understanding its code.

---

# What We Are Looking For

We welcome contributions such as:

* Modern Java examples
* Performance benchmarks
* Documentation improvements
* Bug fixes
* Test improvements
* Architectural experiments
* Engineering Decision Records (EDRs)
* AI-assisted development workflows
* Build and automation improvements

Every contribution should solve a real engineering problem.

---

# What We Avoid

Please avoid submitting:

* CRUD applications
* Boilerplate code
* Low-quality code snippets
* Unexplained examples
* Experimental code without documentation
* AI-generated code that has not been reviewed

Every contribution should answer the question:

> **Why is this useful for a software engineer?**

---

# Engineering Workflow

Each module should follow this workflow:

1. Identify the engineering question.
2. Research authoritative sources.
3. Build a minimal implementation.
4. Add automated tests.
5. Measure performance (when applicable).
6. Document trade-offs.
7. Review AI-generated code.
8. Submit the Pull Request.

---

# Pull Request Guidelines

A Pull Request should be focused on a single topic.

Please include:

* Clear description
* Motivation
* Engineering rationale
* References
* Screenshots (if applicable)
* Benchmark results (if applicable)

Small Pull Requests are preferred over large ones.

---

# Documentation

Every new module should include:

* README
* Theory
* Working examples
* Automated tests
* Engineering decision record
* References

Documentation is considered a first-class citizen.

---

# Code Quality

All contributions must:

* Compile successfully
* Pass the CI pipeline
* Follow formatting rules
* Pass static analysis
* Include tests
* Be understandable
* Be maintainable

Readable code is preferred over clever code.

---

# References

Whenever possible, use authoritative references such as:

* OpenJDK JEPs
* Official Java documentation
* Official Spring documentation
* RFCs
* Academic papers
* Well-established books

Always prefer primary sources over blog posts.

---

# AI-Assisted Contributions

AI tools are welcome.

However:

* Review every generated line.
* Verify every technical statement.
* Never submit code you do not understand.
* Document significant AI assistance when relevant.

AI should improve engineering—not replace engineering judgment.

---

# Philosophy

This project values:

* Curiosity over certainty.
* Simplicity over complexity.
* Understanding over memorization.
* Quality over quantity.

The objective is to create a repository that developers can trust.

Thank you for helping build it.
